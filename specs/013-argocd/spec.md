# Spec 013: ArgoCD GitOps Deployment

## Problem

All k3d deployments are imperative — `helm upgrade --install` from Taskfile tasks. There's no reconciliation loop, no drift detection, and no audit trail for what's deployed. If someone manually edits a k8s resource, nobody notices. If a Helm release drifts from its values file, it stays drifted until the next manual `helm upgrade`.

ArgoCD provides declarative GitOps: define desired state in git, ArgoCD reconciles continuously. This gives us:
- Automatic sync from git commits → cluster state
- Drift detection and self-healing
- Deployment audit trail (who deployed what, when)
- Multi-environment promotion (dev → stage → prod) via git branches or directories
- Dashboard visibility into all deployed applications

## Dependencies

- **Spec 011** (shipped): Helm charts for genai stack exist in platform_monorepo `charts/`
- **Spec 012** (draft): GitLab CE in k3d — ArgoCD pulls from this GitLab instance
- k3d shared cluster `mewtwo` with ingress-nginx

## Requirements

### FR-001: ArgoCD Installation via Helm
Deploy ArgoCD to `platform` namespace using the official `argo/argo-cd` Helm chart. Configuration:
- Server exposed via Ingress at `argocd.platform.127.0.0.1.nip.io`
- Disable Dex (no SSO needed for local dev)
- Admin password from k8s Secret (generated by `task init-secrets` or Helm values)
- Server running in insecure mode (HTTP, no TLS — local dev behind ingress-nginx)
- Redis bundled (default)
- Repo server with Git credentials for local GitLab

### FR-002: GitLab Repository Connection
Configure ArgoCD to connect to local GitLab CE (`http://gitlab.platform.127.0.0.1.nip.io`) as a repository source. Must:
- Use HTTPS (via ingress) or in-cluster Service DNS
- Authenticate with GitLab PAT stored as k8s Secret
- Support both platform_monorepo and genai-mlops repos

### FR-003: ArgoCD Application for genai-infra
Create an ArgoCD Application resource that manages the `genai-infra` Helm release:
- Source: GitLab repo, path `charts/genai-infra`
- Destination: k3d cluster, namespace `genai`
- Sync policy: manual (local dev — don't auto-sync on every commit)
- Values files: `values.yaml` + `values-k3d.yaml`

### FR-004: ArgoCD Application for genai-apps
Create an ArgoCD Application resource that manages the `genai-apps` Helm release:
- Source: GitLab repo, path `charts/genai-apps`
- Destination: k3d cluster, namespace `genai`
- Sync policy: manual
- Depends on genai-infra being healthy (sync wave or health check)

### FR-005: ArgoCD Application for GitLab CE (Spec 012)
Once Spec 012 ships, create an ArgoCD Application for GitLab CE Helm chart:
- Source: GitLab repo (chicken-and-egg: bootstrap via CLI, then ArgoCD manages itself)
- Destination: `platform` namespace
- Sync policy: manual

### FR-006: AppProject Isolation
Create two ArgoCD AppProjects:
- `genai` — allowed namespaces: `genai`. Source repos: genai-mlops, platform_monorepo. Allowed cluster resources: Deployments, StatefulSets, Services, Ingresses, ConfigMaps, Secrets, PVCs.
- `platform` — allowed namespaces: `platform`. Source repos: platform_monorepo. Same resource types plus CRDs.

### FR-007: Taskfile Integration
Add `argocd.yml` to `taskfiles/`:
- `argocd:deploy` → Install ArgoCD via Helm
- `argocd:teardown` → Uninstall
- `argocd:status` → Show Application sync status
- `argocd:sync` → Trigger manual sync for specified app
- `argocd:password` → Extract admin password
- `argocd:open` → Print ArgoCD URL

### FR-008: CLI Access
Install `argocd` CLI and configure login to local instance. Add to `task doctor` checks.

## Acceptance Scenarios

### SC-001: Fresh Install
Given a clean k3d cluster, `task argocd:deploy` installs ArgoCD accessible at `http://argocd.platform.127.0.0.1.nip.io` within 3 minutes.

### SC-002: Application Sync
Given ArgoCD Application for genai-infra, clicking "Sync" in the UI (or `argocd app sync genai-infra`) deploys the Helm release to the `genai` namespace.

### SC-003: Drift Detection
Given a deployed genai-infra release, manually deleting a pod shows "OutOfSync" in ArgoCD within 30 seconds and auto-heals if self-heal is enabled.

### SC-004: Git Commit → Sync
Given a values file change committed to GitLab, ArgoCD detects the change within 3 minutes (polling interval) and shows "OutOfSync" in the UI.

## Non-Functional Requirements

### NFR-001: Resource Budget
ArgoCD services total: ~500MB RAM, 0.5 CPU (server + repo-server + application-controller + Redis).

### NFR-002: Bootstrap Order
1. k3d cluster up (`task k3d:cluster-up`)
2. ArgoCD installed (`task argocd:deploy`)
3. GitLab CE deployed (either via ArgoCD or direct Helm for bootstrap)
4. ArgoCD Applications created for genai stack
5. Sync genai-infra → genai-apps (ordered)

### NFR-003: Fallback
All Helm CLI tasks (`genai-helm:deploy`, `gitlab:deploy`) continue to work without ArgoCD. ArgoCD is additive, not required.

## Architecture

```
                    ┌─────────────────┐
                    │   GitLab CE     │
                    │   (git repos)   │
                    └────────┬────────┘
                             │ git poll (3min)
                    ┌────────▼────────┐
                    │    ArgoCD       │
                    │   (platform)    │
                    └────────┬────────┘
                             │ reconcile
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ genai-   │  │ genai-   │  │ gitlab-  │
        │ infra    │  │ apps     │  │ ce       │
        │ (genai)  │  │ (genai)  │  │(platform)│
        └──────────┘  └──────────┘  └──────────┘
```

## Open Questions

1. **Chicken-and-egg**: GitLab CE runs in k3d, ArgoCD pulls from GitLab. Bootstrap requires deploying GitLab first via Helm CLI, then ArgoCD manages it going forward. Is this acceptable?
2. **Auto-sync vs manual**: For local dev, manual sync is safer (prevents surprise deployments during active development). Should we offer a `--auto-sync` flag in Taskfile?
3. **Image updater**: Should we add ArgoCD Image Updater to auto-detect new image tags pushed to the registry? Or is that overkill for local dev?
